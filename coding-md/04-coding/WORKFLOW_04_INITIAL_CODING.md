# 工程 4: コーディング

## 概要

Figma MCP 経由でデザインを取得し、**セクションごとに** HTML と SCSS を順番にコーディングします。各セクションで必要な部分だけを取得することで、コンテキストを適切なサイズに保ちます。

**工程 3（コンポーネント検出）を実施している場合**: `coding-md/mcp-log/{ページ名}/component-list.md` のコンポーネント一覧を参照し、**共通コンポーネントを先に実装してから、各ページを include で組み立てる**流れにします。全体を一度書いてから共通パーツを抜き出して置き換える手間を避けます。

## 前提条件

- Figma MCP が接続されていること
- Figma で対象のデザインが選択されていること
- 工程 3 を実施した場合は、`component-list.md` が作成されていること

## 重要

- **position:absolute; の使用は原則禁止**
- **要素の並び順や位置は、Figma のノード情報（get_design_context で取得される位置情報）ではなく、スクリーンショットの見た目を参照して組み立ててください**
- Flexbox や Grid などの通常のレイアウト手法を使用し、実際の表示結果を基準にコーディングしてください

## 手順

### 1. Figma からデザイン構造を確認

まず、全体の構造を把握するために、Figma からデザイン構造を取得します：

```bash
# MCPツールを使用（自動実行）
# get_design_context を使用して全体の構造を取得
```

**重要: 画像取得について**

- **画像取得の指示がない限り、画像は取得しない**
  - デフォルトでは、`get_design_context` を使用する際も **画像は取得しません**
  - get_design_context を呼び出す際は dirForAssetWrites を指定しない
  - コーディング指示時は、画像取得について明記しない限り、デザイン情報（フォント、カラー、レイアウト、テキスト内容など）のみを取得してください
  - 新しい画像が必要な場合のみ、明示的に「○○の画像を取得してください」と指示してください
  - `coding-md/mcp-log/{page}/design-context.md` に「画像一覧」セクションがあれば、それらは工程 2 で既に取得済みです（再取得は不要）

**注意**: この時点では全体構造の把握のみを行い、各セクションの詳細は後で取得します。

#### 1-1. MCP ログの保存（必須）

`get_design_context` / `get_variable_defs` で取得した値は、**必ず保存**します。

- 保存先: `coding-md/mcp-log/`
- ルール: `coding-md/mcp-log/README.md` を参照

AI に依頼する場合も、取得後に自動保存されます。

#### インスタンス（コンポーネント）から情報を取得する場合

Figma のコンポーネントインスタンス（例: `h2_02` などのセクションタイトルコンポーネント）から情報を取得する際は、以下の点に注意してください：

1. **`get_metadata` の制限**

   - `get_metadata` では、インスタンスの構造（位置、サイズ、ノード ID）のみ取得できます
   - **インスタンス内のテキスト内容は取得できません**

2. **`get_design_context` を使用**

   - インスタンス内のテキスト内容を取得するには、`get_design_context` を使用します
   - `get_design_context` を使用する際は、**`dirForAssetWrites` パラメータを必ず指定**してください（ワークスペースのルートパスを指定）
   - 例: `dirForAssetWrites: "/Users/username/project-name"`

3. **取得手順**

   ```
   1. Figma で該当のインスタンスノードを選択
   2. ノードIDを確認（例: "2084:12631"）
   3. get_design_context を実行（dirForAssetWrites を指定）
   4. 返されたコードからテキスト内容を抽出
   ```

4. **注意点**
   - `get_design_context` は React + Tailwind のコードを返しますが、これは参考情報として使用
   - 実際のコーディングでは、取得したテキスト内容をプロジェクトの規約に従って実装
   - 画像アセットは工程 2 で既にダウンロード済みのため、`get_design_context` で取得した画像パスは使用しない

### 2. セクションごとにコーディング

各セクションを順番に、**HTML → SCSS の順**でコーディングします：

**注意**: セクション名はプロジェクトごとに異なります。Figma のデザイン構造に従って、必要なセクションをコーディングしてください。

一般的なセクションの種類（プロジェクトによって名称や構成が異なります）：

- ヘッダーセクション
- ファーストビュー（ヒーロー）セクション
- 検索・フィルターセクション
- ニュース・お知らせセクション
- ピックアップ・特集セクション
- マップ・位置情報セクション
- スライダー・カルーセルセクション
- ギャラリー・写真セクション
- 情報・お役立ちセクション
- フッターセクション

#### 各セクションのコーディング手順

**セクション 1: {セクション名}**

1. **Figma から該当セクションのデザインを取得**

   - Figma で該当セクションのノードを選択
   - `get_design_context` を使用して該当セクションのみを取得
   - **重要**: 各セクションのノードを個別に選択して、テキスト内容やレイアウトを正確に確認すること
   - **重要: 画像取得について**
     - **画像取得の指示がない限り、画像は取得しない**
     - `dirForAssetWrites` は指定しない（画像は工程 2 で既にダウンロード済み）
     - コーディング指示時は、画像取得について明記しない限り、デザイン情報（フォント、カラー、レイアウト、テキスト内容など）のみを取得してください
     - 新しい画像が必要な場合のみ、明示的に「○○の画像を取得してください」と指示してください
     - `coding-md/mcp-log/{page}/design-context.md` に記載されている既取得画像は再取得しない

2. **HTML の作成**

   - `{メインHTMLファイル}` に該当セクションの HTML を追加
   - HTML 規約を遵守（後述）

3. **SCSS の作成**

   - `{SCSSモジュールディレクトリ}` に該当セクションの SCSS ファイルを作成
   - SCSS 規約を遵守（後述）

4. **確認**
   - HTML の構造が正しいか確認
   - SCSS が正しくコンパイルされるか確認
   - ブラウザで表示を確認
   - **Playwright でスクリーンショットを取得して確認**（後述）

**セクション 2 以降も同様に繰り返す**

### 3. HTML 規約・SCSS 規約

各セクションの HTML / SCSS は、以下の規約に従って作成します。詳細・チェック項目・修正例は各ドキュメントを参照してください。

- [HTML チェック](./WORKFLOW_04_HTML_CHECK.md) — セマンティック、alt、リンク、BEM、**共通パターンとクラス統一**（「### 9」）、**セクションと BEM の対応**（「### 10」）
- [SCSS チェック](./WORKFLOW_04_CSS_CHECK.md) — BEM、ネスト禁止、rem()、@use、メディアクエリ、変数、**レスポンシブ設計ルール（詳細は同ドキュメント「### 5. レスポンシブ対応」内）**

**ファイル配置**:

- **共通コンポーネント**: `src/ejs/components/` と `src/sass/components/`（工程 3 の component-list.md に従って実装）
- **ページ固有のスタイル**: `src/sass/pages/{ページ名}/` に配置
- **メイン HTML ファイル**: プロジェクトによって名前が異なる場合があります（例: `index.html`, `{ページ名}.ejs` など）

#### 画像アセット配置ルール（`src/public/assets/img/`）

- **共通画像**は `img/common/` に配置する（ロゴなど複数ページで使うもの）
- **ページ専用画像**は `img/{page}/` に配置する（例: `img/top/`, `img/company/`, `img/services/`）
- 新規画像は原則として **ページ専用フォルダ**に置き、2ページ以上で再利用が確定した時点で `img/common/` へ移動する
- 画像を移動・改名した場合は、必ず以下をセットで実施する
  1. 参照パス更新（HTML / テンプレート / SCSS）
  2. `npm run build` で生成物へ反映
  3. 該当ページを目視確認（404 や表示崩れがないことを確認）

### 4. Playwright によるスクリーンショット確認

コーディング後、Playwright を使用してブラウザのスクリーンショットを取得し、Figma デザインと比較確認します。

#### スクリーンショットスクリプトの使用方法

```bash
# 基本的な使用方法（SPビューポートでフルページキャプチャ）
node coding-md/screenshot.cjs

# 特定セクションのみキャプチャ（CSSセレクタを指定）
node coding-md/screenshot.cjs --selector ".l-footer" --viewport sp
node coding-md/screenshot.cjs --selector ".l-header" --viewport pc

# SP/PC両方をキャプチャ
node coding-md/screenshot.cjs --selector ".l-footer" --viewport both

# 出力ファイル名を指定
node coding-md/screenshot.cjs --selector ".l-footer" --name footer

# Figmaデザインと比較
node coding-md/screenshot.cjs --selector ".l-footer" --compare coding-md/mcp-log/design.png
```

#### オプション一覧

| オプション | 説明 | デフォルト値 |
|------------|------|-------------|
| `--url` | 対象URL | 自動検出 |
| `--selector` | キャプチャするCSSセレクタ | なし（全体） |
| `--viewport` | ビューポート（sp, pc, both） | sp |
| `--output` | 出力ディレクトリ | 自動設定 |
| `--name` | 出力ファイル名のプレフィックス | screenshot |
| `--compare` | 比較対象のFigmaデザイン画像パス | なし |

#### セレクタの指定

`--selector` オプションでCSSセレクタを直接指定できます：

```bash
# セレクタを直接指定
node coding-md/screenshot.cjs --selector ".l-footer" --viewport sp
node coding-md/screenshot.cjs --selector "#main-content" --viewport pc
```

**注意**: プロジェクトごとにセクションのクラス名は異なります。実際のHTML構造を確認して適切なセレクタを指定してください。

#### 確認フロー

1. **問題が報告された場合**: まずスクリーンショットを取得して現状を確認
2. **修正後**: 再度スクリーンショットを取得してビフォー/アフターで比較
3. **Figmaデザインとの比較**: `--compare` オプションで並べて確認

#### 自動検出機能

スクリプトは以下を自動検出します：

- **開発サーバーURL**: `package.json`、`gulpfile.js` などからポートを推測し、起動中のサーバーを自動検出
- **出力ディレクトリ**: スクリプトの場所を基準に自動設定

#### 注意事項

- 開発サーバーが起動している必要があります
- サーバーが検出されない場合は `--url` オプションで直接指定してください

## 注意事項

- 指示を待たずに最後まで実行すること
- 画像パスは工程 2 で命名済みのファイル名を使用すること（仮のパスは使用しない）
- 工程 2 でリネームした画像ファイル名を参照して、正しいパスでコーディングすること
