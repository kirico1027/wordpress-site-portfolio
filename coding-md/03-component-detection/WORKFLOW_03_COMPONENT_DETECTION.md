# 工程 3: コンポーネント検出

## 概要

コーディングに入る前に、Figma デザイン（および複数ページがある場合はそれら）から**再利用可能な UI パーツ（コンポーネント）**を検出し、一覧化します。この工程を挟むことで、工程 4 では「いったん全体を書いてから共通パーツを抜き出す」のではなく、**コンポーネントを先に実装し、それを組み合わせてページを組み立てる**流れにできます。

> **工程 0 との関係**  
> 本工程の前に [工程 0: プロジェクトキックオフ](../00-project-kickoff/WORKFLOW_00_PROJECT_KICKOFF.md) を実施すること。  
> 工程 0 で作成した `commonization-plan.md`（レイアウト基準・共通パーツ・partials 構成）をベースに、本工程で `component-list.md` へ詳細化する。

## 目的

- **手戻りを減らす**: 全体コーディング → 共通パーツ認識 → コンポーネント化 → 各ページへの置き換え、という二度手間を避ける
- **一貫性を保つ**: 最初からコンポーネント単位で設計することで、命名や構造のブレを防ぐ
- **工数削減**: 複数ページで使うパーツを一度だけ実装し、include で使い回す

## 前提条件

- [工程 0（プロジェクトキックオフ）](../00-project-kickoff/WORKFLOW_00_PROJECT_KICKOFF.md) が完了し、`commonization-plan.md` が作成されていること
- Figma MCP が接続されていること
- 工程 1（フォント・カラー）が完了していること
- 工程 2（画像エクスポート）は、画像を使うコンポーネントがある場合に実施済みであること（必須ではない）

## 出力物

- **コンポーネント一覧**: `coding-md/mcp-log/{ページまたはプロジェクト名}/component-list.md`
- 既存コンポーネントがある場合は、`src/ejs/components/` や `src/scss/components/` の構成を参照しつつ、**新規・要追加**のコンポーネントを一覧に含める

## 手順

### 1. 対象範囲の確認

- コーディング対象が **1 ページ** か **複数ページ** かを確認する
- 複数ページがある場合: 全ページのデザイン（または主要ページ）に目を通し、同じ見た目・同じ役割のブロックを把握する

### 2. デザイン上のブロックの洗い出し

Figma の画面・ノード構造を頼りに、次の観点でブロックを洗い出します。

- **レイアウトの塊**: ヘッダー、フッター、メインの各セクション、サイドバーなど
- **繰り返し出現するパターン**: カード、リスト項目、見出し＋説明のブロック、CTA ボタンなど
- **デザインは違うが役割が同じもの**: 例）「トップのカード」と「一覧ページのカード」が別デザインなら、別コンポーネント or モディファイアで表現するかを検討

#### チェックポイント

- 同じような見た目・構造が 2 回以上出てくるか
- 別ページでも同じブロックが使われそうか
- テキスト・画像だけ差し替えて使い回せそうか

### 3. 共通コンポーネントとページ固有の切り分け

洗い出したブロックを次のように分類します。

| 分類 | 説明 | 例 |
|------|------|-----|
| **共通コンポーネント** | 複数ページ・複数箇所で使い回す。EJS の include や共通 SCSS として実装する | ヘッダー、フッター、パンくず、ページヘッダー、カード、TOC、関連記事ブロック |
| **ページ固有** | そのページ・そのセクションだけで使う。ページ用 HTML と `src/sass/pages/` 配下の SCSS に記述する | トップ専用のヒーロー、採用一覧ページ専用のフィルター UI |

- **共通コンポーネント**は、工程 4 で「コンポーネントの HTML（EJS）と SCSS」を先に作り、その後で各ページから include して組み立てる（配置: `src/ejs/components/` と `src/sass/components/`）
- **ページ固有**は、そのページの HTML に記述し、SCSS は `src/sass/pages/{ページ名}/` に配置する

### 4. コンポーネント一覧の作成

次の内容を `coding-md/mcp-log/{ページまたはプロジェクト名}/component-list.md` にまとめます。

```markdown
# コンポーネント一覧

## 既存コンポーネント（流用）

| コンポーネント名 | 使用箇所 | 備考 |
|------------------|----------|------|
| header           | 全ページ |      |
| footer           | 全ページ |      |
| breadcrumb       | 下層ページ |     |

## 新規作成するコンポーネント

| コンポーネント名（ブロック名） | 使用箇所 | 想定パラメータ・バリエーション | 備考 |
|-------------------------------|----------|--------------------------------|------|
| pageHeader                    | 一覧系ページ | icon, enText, title, description, className | 余白は className でページ側指定 |
| card2                         | トップ・人を知る一覧 | link, image, alt, title, tags[], category, date | インタビューカード |
| （ページ固有）people-filter   | 人を知る一覧のみ | - | 絞り込み UI、コンポーネント化しない |
```

- **既存コンポーネント**: すでに `src/ejs/components/` などにあるもの。流用するだけなので「使用箇所」を書いておく
- **新規作成**: これから EJS + SCSS で作るもの。**想定パラメータ・バリエーション**を書いておくと工程 4 で実装しやすい
- **ページ固有**も一覧に「コンポーネント化しない」と明記しておくと、工程 4 で「include にしない」判断の根拠になる

### 5. 既存プロジェクトの場合の差分確認

- 既に `src/ejs/components/` や `src/scss/components/` がある場合は、その中身を確認する
- 一覧の「新規作成」には、**まだ存在しないコンポーネント**だけを書く
- 既存コンポーネントで「デザイン変更でパラメータが足りない」ようなら、一覧の備考に「既存 ○○ の拡張」と書いておく

### 6. 工程 4 への引き継ぎ

- 工程 4 のコーディングでは、**この component-list.md を参照する**
- 手順の順序は次のいずれかとする:
  - **A. コンポーネント優先**: 一覧の「新規作成」を先に実装 → その後、各ページを「既存＋新規コンポーネントの include ＋ ページ固有」で組み立てる
  - **B. ページ単位だがコンポーネントを前提**: 各ページをコーディングする際、そのページで使うブロックが一覧の「共通」なら include で組み立て、なければ新規コンポーネントを追加してから include する

プロジェクト方針に合わせて A/B を選び、工程 4 のドキュメントまたは README に「コンポーネント一覧に従い、コンポーネント中心でコーディングする」と明記することを推奨します。

## 保存ルール

- 出力ファイル: `coding-md/mcp-log/{ページまたはプロジェクト名}/component-list.md`
- 複数ページをまとめて見る場合は、プロジェクト名や `all-pages` などのフォルダ名でよい
- MCP ログの保存ルールは `coding-md/mcp-log/README.md` に従う

## 注意事項

- コンポーネントの「名前」は、既存の命名規則（例: BEM のブロック、`c-card` など）に合わせる
- 無理に共通化しない: 似ていても仕様が大きく違うなら、別コンポーネント or ページ固有としておく
- 一覧は「完璧な設計」でなくてよい。工程 4 で実装しながら「ここは共通化する」と判断が変わっても、その時点で一覧を更新すればよい

## 参考: コンポーネント検出の例

- **パンくず**: 全下層ページで同じ構造 → 共通（items を渡す）
- **ページタイトル＋説明**: 一覧系で何度も同じレイアウト → pageHeader として共通
- **カード（画像＋タイトル＋テキスト＋リンク）**: トップと一覧で同じ見た目 → 共通 card / card2
- **記事詳細の「INDEX」ブロック**: 詳細ページで 1 回だけだが、他詳細ページでも流用 → toc として共通
- **「事例を絞り込む」のような検索 UI**: 人を知る一覧だけで使う → ページ固有（コンポーネント化しない、または後から必要なら共通化）
