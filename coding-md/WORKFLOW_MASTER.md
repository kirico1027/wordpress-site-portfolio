# Figma MCP コーディングワークフロー（全体）

## 概要

Figma MCP を使用した Web サイトのコーディング工程をまとめたマスターファイルです。各工程の詳細は個別の md ファイルを参照してください。

## 前提条件

- Figma MCP が接続されていること
- Figma で対象のデザインが選択されていること
- プロジェクトのセットアップが完了していること（Gulp、SCSS 環境など）

## 全工程共通ルール

すべての工程に共通する前提ルールです。各工程を実施する前に確認してください。

### レイアウト実装ルール（高さ指定）

- **テキストを含む要素は、固定 `height` を原則使用しません**
- 高さ調整は `padding`（必要に応じて `min-height`）で実装し、コンテンツ量や画面幅の変化に追従できるようにします
- 固定 `height` は、装飾目的で内容量が増減しない要素など、必要性が明確な場合に限定します

### レイアウト実装ルール（縦方向の余白）

- **上下の間隔調整は原則 `margin-top` で統一します**
- 前要素に `margin-bottom` は付けず、後続要素に `margin-top` を付与して間隔を管理します
- 同階層で隣り合う要素同士（例: 見出しブロックと一覧ブロック）の間隔も、上要素の `padding-bottom` ではなく下要素の `margin-top` で管理します
- 外側の要素間の距離調整は `margin` で行い、コンテンツ内側の余白調整は `padding` で行います
- これにより、余白の責務を後続要素側に集約し、コンポーネントの再利用時に余白衝突を減らします
- **`margin-bottom` は付けません**

### レスポンシブ設計（方針）

- 固定幅に頼らず、画面幅に応じて自然に伸縮する CSS を優先し、メディアクエリは最小限にとどめる。
- 単位の使い分け、レイアウト、`max-width`、ブレイクポイントの役割分け、テキスト幅・改行の扱いなどの**詳細ルール**は、[工程 4: SCSS チェック](./04-coding/WORKFLOW_04_CSS_CHECK.md) の「### 5. レスポンシブ対応」→「レスポンシブ設計ルール（詳細）」を参照する。

### MCP ログの保存（必須）

- Figma MCP で取得したデザイン情報（フォント、カラー、レイアウト、テキスト内容など）は、必ず `coding-md/mcp-log/` ディレクトリに保存してください
- 保存ルール: [coding-md/mcp-log/README.md](./mcp-log/README.md) を参照
- これにより、後続の品質チェックやコードとの整合性確認時に、MCP に再接続することなく参照できます

### 画像取得について

- **画像取得の指示がない限り、画像は取得しません**
- `get_design_context` を使用する際も、デフォルトでは画像は取得せず、デザイン情報（フォント、カラー、レイアウト、テキスト内容など）のみを取得してください
- 新しい画像が必要な場合のみ、明示的に「○○の画像を取得してください」と指示してください
- 詳細: [coding-md/mcp-log/README.md](./mcp-log/README.md) の「画像取得のベストプラクティス」を参照

### HTML / CSS クラス設計ルール（統一指示）

HTML / CSS は、再利用性・可読性・保守性を重視して設計してください。
クラス名は「見た目」ではなく「役割・構造・パーツ」が一目で分かる命名にします。

#### 基本方針

- BEM をベースに設計する
  - Block: `voice-card`
  - Element: `voice-card__head`
  - Modifier: `voice-card--featured`
- 命名は最終的に BEM へ統一する（新規実装は BEM 必須）。既存改修の移行期間のみハイフン命名を暫定許容するが、同一コンポーネント内で命名方式を混在させず、ブロック単位で統一する
- `.section .btn` のような親要素に依存したセレクタは使用しない
- CSS は単体クラスに対して記述し、詳細度を低く保つ
- 「共通パーツ」と「文脈（ページ・セクション）」を分離する

#### クラスの役割分担

- 共通パーツ（再利用前提）
  - 複数箇所で使うものは汎用クラスとして定義する
  - 例: `btn`, `section-title`, `section-subtitle`, `section-heading`, `section-heading-row`, `container`, `voice-card`, `voice-cards`
- 文脈クラス（ページ・セクション固有）
  - 特定の場所だけのレイアウトや余白調整に使用する
  - 例: `archive-voice__cards`, `careers__layout`, `news__subtitle`, `careers__subtitle`
- Modifier（見た目・状態）
  - 色・サイズ・状態の違いに使用する
  - 例: `btn--dark`, `btn--light`, `section--dark`, `is-active`

#### セクションと子要素の BEM 対応（名前付きブロック）

- **セクションにブロック名を付ける**: そのエリアをひとまとまりのブロックとして扱うときは、`<section>` にページ共通の `section` / `section--*` に加えて **`services`, `news`, `careers`, `section-cta` などのブロック名**を付ける（例: `class="section section--light news"`）。
- **セクション直下のレイアウトは `{block}__*` に寄せる**: 一覧グリッドやワイド用ラッパーは、セクション名だけ変わる別ブロック（`news-grid`）を増やさず **`news__grid`, `services__grid`, `services__wide`** のように **親ブロックのエレメント**で表す。
- **カード・パーツは別ブロックのままでよい**: 複数セクションで再利用する単位は **`news-card`, `service`, `btn`** のように独立ブロック名のままとし、親セクションとの関係は **ネストとクラスの組み合わせ**で表す。
- **カード内の役割ごとにエレメントクラスを付ける**: 記事タイトル・本文など意味が違うテキストには **`news-card__title`** のようにクラスを付与する。**`.news-card p` のようなタグ結合だけに頼ってスタイルを当てない**（ワークフローのタグ結合回避と整合させる）。
- **`○○__title` が付いた要素があるブロック**は、親に **`○○` ブロッククラス**があることを確認する（例: `section-cta__title` なら `<section class="… section-cta">`）。

#### 共通パターンの検出とクラス名の統一

- **同じ役割・同じ DOM 構造**が複数セクション・複数ページに現れたら、**共通クラス（共通パーツのブロック）にまとめることを検討する**。実装時・レビュー時に見つけて揃える運用を基本とし、無理な事前設計はしない。
- **統一する単位**は「意味とマークアップのかたまり」にする（例: 英語見出し＋サブタイトルの塊、それに VIEW MORE が付いた横並びの一行）。**細かい見た目の差**は、共通ブロックに **文脈クラス**（`{section}__subtitle` など）や **Modifier** を足して吸収する。
- **同一パターンに、セクション名だけが違う別ブロック名のラッパーを増やし続けない**（例: 見出し行＋「VIEW MORE」が同型なら `section-heading-row` のように共通ラッパーへ寄せる）。
- **共通化の粒度**はプロジェクトでよいが、「2 回目で検討・3 回目で共通化」など基準を決めておくとブレにくい。**すべてのパターンをワークフローに列挙しない**。新しい共通パターンが増えたら、`component-list.md` やプロジェクトのコンポーネント一覧に追記する運用でもよい。
- **詳細なチェック項目**は [工程 4 補足: HTML チェック](./04-coding/WORKFLOW_04_HTML_CHECK.md) の「共通パターンとクラス統一」「セクションと BEM の対応」を参照する。

#### 複数クラスの使い方

- 再利用パーツ + 文脈調整のために、必要な場合のみ複数クラスを付与する
- 同一要素に、同じ責務（同じプロパティ群）を持つクラスを重複付与しない
  - 例: `section__sub` と `section-subtitle` のように同じ `font-size` / `margin` を持つクラスの併用は避ける
- 補足:
  - 複数クラスの付与やコンポーネント分離は、再利用性や明確な役割分離が必要な場合にのみ行う
  - 小規模な要素や一度しか使わない箇所では、過剰にクラスを分割しない

#### セクション（ページ固有ブロック）実装の原則

**名前付きブロック（BEM のブロック名でまとまる領域）**を実装するときの共通指針です。ブロック名はページやデザインに合わせて決め、以下は **`{block}` をその名前に読み替えて**適用する。

- **プレフィックスの一貫性**: そのブロック専用のパーツは **`{block}__{element}`**（BEM の要素）で命名する。レイアウト用ラッパーにも `__layout` / `__content` など役割が分かる名前を付ける。
- **無名ラッパーを避ける**: スタイルの当て先となる `div` 等にはクラスを付与し、`.block > div:first-child` のような **位置・タグ依存のセレクタ**を使わない。
- **タグ結合を避ける**: `.block h1`、`.block p` のようにブロック + タグで狙わず、**要素クラス**（例: `{block}__title`）に寄せる。
- **共通クラスとの役割分担**: サイト全体で繰り返すパターンは **共通ブロック**（プロジェクトで定義した見出し・ボタンなど）を使い、レイアウトや当該ブロック固有の調整は **`{block}__*`** に書く。装飾の差分は **同一ブロック名のモディファイア**で表現し、別名空間のクラスを無計画に併用しない。
- **共有しないなら汎用クラスを重ねない**: 当該ブロックだけのビジュアルなら、任意の汎用カードクラスに頼らず **`{block}__img` のように一要素へスタイルを集約**する。複数箇所で同じ見た目を使い回す場合のみ共通ブロック＋モディファイアを検討する。
- **セレクタの詳細度**: `.block__layout > .block__content` のように親子結合でスコープを狭める必要がなければ、**単体の `.block__content` に寄せる**（クラス名がページ内で十分一意であることが前提）。再利用・コピー時はクラス名の意味範囲に注意する。
- **見出しの幅**: カラム内の見出しは **親幅を活かす**のを基本とし、レスポンシブを損ねないよう **不要な `max-width` + `white-space: nowrap` で幅だけ切り詰めない**（改行はマークアップまたは通常の折り返しに任せる）。

#### 画像の表現（CSS 背景と `<img>`）

- **役割で選ぶ**。見た目の実装のしやすさだけで `background-image` と `<img>` のどちらかに固定しない。
- **装飾・雰囲気・レイアウト用**で、画像を除いても伝えるべき情報が失われない場合は、**`background-image` 等を要素に指定する**実装でもよい（セクション脇のビジュアルなど）。
- **コンテンツとして意味がある**場合（説明・証拠、製品や記事の主題、データの可視化など）、**検索・共有・アクセシビリティで画像として扱う**必要がある場合は **`<img>`** を使い、内容に応じた **`alt`** を付ける。
- **判断の目安**: 「画像を除いても主メッセージは成立するか」→ 成立するなら装飾寄り（背景が選択肢）、成立しないなら **`<img>` を優先して検討**する。
- **固定比率のビジュアル枠**: `<img>` に直接 `aspect-ratio` を付けず、`figure` または `__figure` / `__media` などの**親ラッパーで比率を決め**、子 `<img>` は `width: 100%; height: 100%; object-fit: cover` で収める。単一ソースの規約は **`.cursor/rules/markup-bem-css.mdc`**（「画像のアスペクト比」）。チェック項目・マークアップ例は [工程 4 補足: HTML チェック](./04-coding/WORKFLOW_04_HTML_CHECK.md) の「### 4. 画像要素」、SCSS 側は [工程 4 補足: SCSS チェック](./04-coding/WORKFLOW_04_CSS_CHECK.md) の「レイアウト設計」内の画像関連箇条書きを参照する。

例:

```html
<div class="archive-voice__cards voice-cards">
  <article class="voice-cards__card voice-card">
    <div class="voice-card__item">
      <div class="voice-card__head">
        ...
      </div>
    </div>
  </article>
</div>
```

## 工程一覧

### 0. プロジェクトキックオフ（コーディング前整理）

**[工程 0: プロジェクトキックオフ](./00-project-kickoff/WORKFLOW_00_PROJECT_KICKOFF.md)**

- **いきなりコーディングを開始しない**
- トップページ全体を確認し、レイアウト基準（幅・余白・セクション間隔）と共通 UI パーツを整理する
- `commonization-plan.md` に共通化方針を提案してから静的コーディングを開始する
- WordPress 化案件では、固定ページ / 投稿 / CPT / タクソノミー / カスタムフィールド / template-parts 化要素を [WordPress 設計書テンプレート](./07-wordpress-design/WORDPRESS_DESIGN_TEMPLATE.md) に沿って事前整理する
- 工程 3（コンポーネント検出）の入力となる

### 1. フォント・カラー情報の取得

**[工程 1: フォント・カラー変数の準備](./01-font-color-setup/WORKFLOW_01_FONT_COLOR_SETUP.md)**

- Figma からフォントとカラー情報を取得
- MCP ログは `coding-md/mcp-log/` に保存
- Google Fonts CDN の読み込み
- SCSS 変数の登録

### 2. 画像のエクスポート（指示があるときのみ）

**[工程 2: 画像のダウンロードと命名](./02-image-export/WORKFLOW_02_IMAGE_RENAMING.md)**

- **注意**: 画像取得の指示がある場合のみ実施します
- Figma から画像をダウンロード
- `src/public/assets/img/` に保存
- 命名規則に従ってリネーム
- ビルドツールが自動で最適化・出力を行う

### 3. コンポーネント検出

**[工程 3: コンポーネント検出](./03-component-detection/WORKFLOW_03_COMPONENT_DETECTION.md)**

- コーディング前に、デザインから**再利用可能な UI パーツ（コンポーネント）**を検出して一覧化する
- 「共通コンポーネント」と「ページ固有」を切り分け、`coding-md/mcp-log/{ページ名}/component-list.md` に保存する
- 工程 4 では、この一覧を参照し、**コンポーネントを中心にコーディング**する（先にコンポーネントを実装し、各ページは include で組み立てる）
- 全体を一度書いてから共通パーツを抜き出して置き換える手間を避ける

### 4. コーディング

**[工程 4: コーディング](./04-coding/WORKFLOW_04_INITIAL_CODING.md)**

- Figma からデザイン構造を確認
- **工程 3 で作成したコンポーネント一覧（component-list.md）を参照**し、コンポーネント単位でコーディングする
- MCP ログは `coding-md/mcp-log/` に保存（画像取得の指示がない限り、画像は取得しない）
- **共通コンポーネント**は先に EJS・SCSS で実装し、各ページでは include で組み立てる。**ページ固有**はそのページの HTML/SCSS に直接記述する（SCSS は `src/sass/pages/` 配下）
- **セクションごとに** HTML → SCSS の順でコーディング
- 各セクションで必要な部分だけを取得し、コンテキストを適切なサイズに保つ
- 命名済みの画像パスを使用

**実装後の個別チェック（サブドキュメント）**:

- [HTML チェック](./04-coding/WORKFLOW_04_HTML_CHECK.md) — 構造・セマンティック・アクセシビリティ・セクションと BEM の対応
- [SCSS チェック](./04-coding/WORKFLOW_04_CSS_CHECK.md) — BEM・rem()・@use・メディアクエリ・レスポンシブ設計ルール（詳細）
- [全体チェック](./04-coding/WORKFLOW_04_OVERALL_CHECK.md) — HTML/CSS/画像の整合性

### 5. スクリーンショット比較

**[工程 5: スクリーンショット比較と調整](./05-screenshot-comparison/WORKFLOW_05_SCREENSHOT_COMPARISON.md)**

- Playwright で実サイトのスクリーンショットを取得
- Figma MCP でスクリーンショットを取得
- 比較して差異を修正
- 2 回繰り返して精度を上げる

### 6. 品質チェック

**[工程 6: 品質チェック（エディタ内で完結）](./06-quality-check/WORKFLOW_06_QUALITY_CHECK.md)**

- タイポグラフィ、色、画像、JS/アニメーション、フォーム、meta/head、HTML/リンクのコード上の確認
- MCPログ（`coding-md/mcp-log/{page}/design-context.md`）を参照して、実装コードとの整合性を確認

## 工程の順序

```
全工程共通ルール（MCPログの保存：必須）
    ↓
工程0（プロジェクトキックオフ：トップ確認・共通化方針・WP設計）
    ↓
工程1（フォント・カラー情報の取得）
    ↓
工程2（画像のエクスポート：指示があるときのみ）
    ↓
工程3（コンポーネント検出：共通パーツの一覧化）
    ↓
工程4（コーディング：コンポーネント中心でHTML→SCSS）
    ↓
工程5（スクリーンショット比較と調整 × 2回）
    ↓
工程6（品質チェック：エディタ内で完結）
```

## 各工程は独立した作業としても実施可能

上記の順序は推奨フローですが、**各工程は独立した作業としても実施できます**。必ず 1 → 2 → … の順で通す必要はありません。

- **工程 1** … プロジェクト初期に 1 回実施すればよく、単体で可能
- **工程 2** … 画像が必要なタイミングで、対象ページ・対象画像だけ実施可能
- **工程 3** … 単体で可能。工程 4 の前にやると効くが、やらなくても 4 は実施できる（その場合は従来どおりセクション単位でコーディング）
- **工程 4** … 1 ページ・1 セクション単位で独立して実施可能。工程 3 の一覧がなくても、そのページだけコーディングできる
- **工程 5** … 工程 4 で該当箇所を実装したあと、その部分だけ比較する作業として単体で実施可能
- **工程 6** … 実装が終わったファイル・ページ単位で、単体でチェック可能

「このページだけ先にコーディングしたい」「画像は後回しで、まず 3 と 4 だけやりたい」など、**部分的な進め方**も可能です。

## 注意事項

- **新規案件は工程 0 から開始すること**（共通化方針の提案・合意後にコーディングへ進む）
- 各工程は順番に実行すること
- 前の工程が完了してから次の工程に進むこと
- 各工程で問題があれば必ず修正すること
- 画像取得の指示がない限り、画像は取得しないこと
